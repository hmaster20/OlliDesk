# Версии приложения

## 1. Как формировать саму версию (SemVer)

Используйте трехкомпонентный формат X.Y.Z (Мажорная.Минорная.Патч):

* Патч (Z): 0.1.2 — если вы исправили баг (коммиты с префиксом fix).
* Минорная (Y): 0.2.0 — если вы добавили новую фичу, обновили логику, но старый код продолжает работать (feat, add, upd). На этапе 0.Y.Z (до релиза 1.0.0) минорная версия часто означает и ломающие изменения, так как проект активно развивается.
* Мажорная (X): 1.0.0 — первый стабильный публичный релиз. Далее увеличивается только при ломающих изменениях (breaking changes), когда старый код перестанет работать с новой версией библиотеки.

------------------------------

## 2. Шаг за шагом: Процесс выпуска релиза

Представим, что вы находитесь на версии 0.1.1 и хотите выпустить 0.1.2 (или 0.2.0).

### Шаг A: Обновление версии в коде

Версия в вашем pyproject.toml всегда должна совпадать с будущим Git-тегом. Никогда не меняйте её вручную, используйте консольные утилиты.

* Если вы используете Poetry:

poetry version patch   # Изменит 0.1.1 -> 0.1.2# или
poetry version minor   # Изменит 0.1.1 -> 0.2.0

* Если вы используете Hatch:

hatch version patch

* Если у вас простой pyproject.toml (проект на setuptools), версию можно обновить с помощью утилиты bump-my-version или вручную, но строго перед созданием тега.

### Шаг B: Генерация Changelog через git-cliff

Теперь, когда код готов к релизу, вы запускаете git-cliff. Утилита увидит все коммиты add, fix, upd со времени прошлого тега v0.1.1 и создаст описание для новой версии.

#### Генерируем чейнджлог и обновляем CHANGELOG.md

git-cliff --tag v0.1.2 -o CHANGELOG.md

### Шаг C: Фиксация изменений в Git

Добавляем измененный pyproject.toml и CHANGELOG.md в коммит. Этот коммит называют релизным:

git add pyproject.toml CHANGELOG.md
git commit -m "chore: release v0.1.2"

### Шаг D: Создание Git-тега

Тег ставится строго на релизный коммит, который вы только что сделали.

💡 Правило: Всегда используйте префикс v перед номером версии в Git-тегах (v0.1.2). Это стандарт для большинства утилит (включая GitHub/GitLab и git-cliff).

Всегда используйте аннотированные теги (-a), так как они содержат метаданные (кто поставил, когда) и сообщение:

git tag -a v0.1.2 -m "Release v0.1.2"

### Шаг E: Пуш в удаленный репозиторий

Обычный git push не отправляет теги на GitHub/GitLab. Их нужно пушить отдельно:

git push origin main
git push origin v0.1.2  # Пуш конкретного тега# или пуш всех локальных тегов сразу:
git push origin --tags

## Как автоматизировать этот процесс?

Чтобы не выполнять эти 5 шагов вручную и случайно не ошибиться в номерах версий, разработчики используют инструмент release-plz (для Rust) или python-semantic-release (для Python).
Если вы хотите делать это одной командой локально, можно написать простой Makefile или Bash-скрипт release.sh:

```sh
#!/bin/bash
# Использование: ./release.sh minor или ./release.sh patch
# 1. Поднимаем версию в pyproject.toml

poetry version $1
NEW_VERSION=$(poetry version -s)

# 2. Генерируем CHANGELOG

git-cliff --tag "v$NEW_VERSION" -o CHANGELOG.md

# 3. Коммитим изменения

git add pyproject.toml CHANGELOG.md
git commit -m "chore: release v$NEW_VERSION"

# 4. Ставим тег

git tag -a "v$NEW_VERSION" -m "Release v$NEW_VERSION"

echo "Готово! Версия v$NEW_VERSION подготовлена. Выполните: git push origin main --tags"
```

Самый надежный способ избежать путаницы — сделать pyproject.toml единственным источником правды (Single Source of Truth). В коде app.py версия никогда не должна прописываться вручную — Python должен считывать её динамически во время запуска.
Вот как правильно организовать этот процесс для десктопного PyQt/PySide приложения

## Шаг 1. Переносим чтение версии в app.py

В стандартной библиотеке Python (начиная с версии 3.8) есть модуль importlib.metadata. Он умеет вытягивать версию пакета прямо из его метаданных.
Замените хардкод в OlliDesk/core/app.py:

```py
import sysfrom importlib.metadata
import version, PackageNotFoundErrorfrom PyQt6.QtWidgets
import QApplication  # или PySide6

# ... внутри вашего класса или функции инициализации

        self.app = QApplication(sys.argv)

        # Динамически получаем версию из метаданных пакета "ollidesk"
        try:
            current_version = version("ollidesk")
        except PackageNotFoundError:
            # Делбек-вариант на случай, если проект запущен в режиме разработки без установки
            current_version = "0.1.1-dev"

        self.app.setApplicationVersion(current_version)
        self.app.setApplicationName("OlliDesk")
        self.app.setOrganizationName("OlliDesk")
```

Как это работает: Когда ваше приложение установлено (например, через pip install -e . в режиме разработки или собрано в .whl/exe), Python всегда будет знать актуальную версию из pyproject.toml. Вам больше никогда не придется открывать app.py для правки версии

## Шаг 2. Инструмент для автоматического поднятия версий

Чтобы автоматизировать процесс изменения версии в pyproject.toml, обновления CHANGELOG.md через git-cliff и создания тегов, используйте инструмент bump-my-version (современный преемник популярного bumpversion).

## 1. Установка

pip install bump-my-version

## 2. Настройка в pyproject.toml

Добавьте в конец вашего pyproject.toml блок конфигурации. Он свяжет обновление версии с утилитой git-cliff:

```toml
[tool.bumpversion]
current_version = "0.1.1"
parse = "(?P<major>\\d+)\\.(\\d+)\\.(?P<patch>\\d+)"
serialize = ["{major}.{minor}.{patch}"]
search = "version = \"{current_version}\""
replace = "version = \"{new_version}\""# Автоматически коммитить изменения и создавать тег
commit = true
tag = true
sign_tags = false
tag_name = "v{new_version}"
commit_message = "chore(release): bump version {current_version} → {new_version}"
```

### Указываем, в каких файлах искать и менять версию

```toml
[[tool.bumpversion.files]]
filename = "pyproject.toml"
```

### Интеграция с git-cliff: запускаем генерацию чейнджлога прямо в процессе "бампа"

[[tool.bumpversion.hooks]]
setup = "git-cliff --tag v{new_version} -o CHANGELOG.md"

## Шаг 3. Как теперь выглядит процесс релиза (в 1 команду)

Когда вы накопили пул коммитов (add, fix, upd) и готовы выпустить новую версию приложения, вам достаточно выполнить в терминале всего одну команду.

* Если это багфикс (0.1.1 -> 0.1.2):

bump-my-version bump patch

* Если это новая фича или крупное обновление (0.1.1 -> 0.2.0):

bump-my-version bump minor

## Что произойдет под капотом автоматически

   1. Утилита откроет pyproject.toml и изменит version = "0.1.1" на version = "0.1.2".
   2. Запустится встроенный хук: git-cliff отсканирует коммиты от прошлого тега до текущего момента и допишет новые логи в CHANGELOG.md.
   3. Утилита автоматически сделает git add pyproject.toml CHANGELOG.md.
   4. Создаст Git-коммит с текстом chore(release): bump version 0.1.1 → 0.1.2.
   5. Поставит Git-тег v0.1.2 на этот коммит.

Вам останется только отправить изменения на сервер:

git push origin main --tags

* Перед сборкой вы делаете pip install -e . (чтобы зарегистрировать метаданные в локальном окружении).
* Сборщик подтянет корректную динамическую версию, и в скомпилированном .exe или .app файле внутри OlliDesk всегда будут отображаться правильные цифры.

Фиксировать конкретную версию в обработчике ошибки — плохое решение. В активной разработке это значение быстро устареет, и вы снова запутаетесь.
Чтобы сразу видеть проблему в интерфейсе, лучше возвращать явную строку-индикатор ошибки, либо динамически считывать pyproject.toml «на лету», если пакет не установлен.
Вот три отличных варианта решения на выбор:

## Вариант 1. Информативная строка-индикатор (Самый простой)

Вместо фейковой версии возвращайте строку, которая сразу даст вам понять, что вы забыли сделать pip install -e . в текущем виртуальном окружении.

try:
    current_version = version("ollidesk")except PackageNotFoundError:
    # Сразу видно в окне "О программе", что окружение не настроено
    current_version = "unknown-dev (run 'pip install -e .')"

* Плюс: Идеально для дебага. Если в интерфейсе OlliDesk вы видите этот текст — вы сразу понимаете, что нужно запустить в терминале.

------------------------------

## Вариант 2. Динамическое чтение из pyproject.toml (Автоматический)

Если проект запущен из исходников, файл pyproject.toml всегда лежит в корне. Можно научить Python читать версию напрямую из него, если метаданные пакета недоступны.
Для этого в стандартной библиотеке Python (начиная с 3.11) есть модуль tomllib (в Python 3.8–3.10 используется библиотека toml).

```py
import sysfrom pathlib
import Pathfrom importlib.metadata
import version, PackageNotFoundError

try:
    current_version = version("ollidesk")except PackageNotFoundError:
    # Пытаемся найти pyproject.toml относительно текущего файла app.py
    # OlliDesk/core/app.py -> поднимаемся на 2 уровня вверх до корня проекта
    pyproject_path = Path(__file__).resolve().parents[2] / "pyproject.toml"

    if pyproject_path.exists():
        # Для Python 3.11+ встроенный tomllib
        import tomllib
        with open(pyproject_path, "rb") as f:
            data = tomllib.load(f)
            current_version = data.get("project", {}).get("version", "0.0.0-path-err") + "-src"
    else:
        current_version = "0.0.0-unknown"
```

* Плюс: Полная автоматизация. Даже если вы создадите новое venv и запустите python main.py без установки, программа сама залезет в pyproject.toml и выведет, например, 0.1.1-src. Вы всегда будете видеть реальные цифры.

## Вариант 3. Использование модуля _version.py (Классический подход)

Многие крупные библиотеки (например, setuptools_scm) автоматически генерируют служебный файл_version.py во время сборки или бампа версии.
Если вы используете bump-my-version (из предыдущего шага), вы можете заставить его обновлять версию еще и в специальном маленьком файле OlliDesk/core/_version.py:

   1. Добавьте в pyproject.toml:

[[tool.bumpversion.files]]
filename = "OlliDesk/core/_version.py"
search = "__version__ = \"{current_version}\""
replace = "__version__ = \"{new_version}\""

   1. В самом файле OlliDesk/core/_version.py оставьте одну строку:

__version__ = "0.1.1"

   1. В app.py импортируйте её напрямую:

from OlliDesk.core._version import __version__# ...
self.app.setApplicationVersion(__version__)

* Плюс: Никаких try-except блоков, импорт работает всегда и везде, а bump-my-version гарантирует, что эта строка обновится одновременно с pyproject.toml и CHANGELOG.

Какой из вариантов (информативный unknown, авточтение pyproject.toml или автогенерация _version.py) кажется вам наиболее удобным для OlliDesk?

## Ежедневная разработка (Как обычно)

Вы работаете в привычном режиме:

   1. Меняете код OlliDesk.
   2. Пишете коммиты с префиксами: fix: исправил баг в окне, add: добавил кнопку, upd: обновил логику.
   3. Делаете git add, git commit, git push.
   4. Запускаете python main.py для тестирования — приложение всегда видит текущую версию (например, 0.1.1).

## Шаг за шагом: Выпускаем обновление (Хотфикс или Минор)

Когда вы решили, что накопленные изменения пора зафиксировать в новый релиз, вместо привычной цепочки add -> commit -> push вы делаете следующее:

## Шаг 1: Запуск утилиты бампа

Прямо в терминале в корне проекта вы выполняете всего одну команду, в зависимости от типа изменений.

* Если это хотфикс / патч (версия 0.1.1 превратится в 0.1.2):

bump-my-version bump patch

* Если это минорное обновление (версия 0.1.1 превратится в 0.2.0):

bump-my-version bump minor

## Шаг 2: Что происходит автоматически за 1 секунду

Утилита bump-my-version сама делает всю рутину, которую раньше приходилось контролировать головой:

   1. Заходит в pyproject.toml и меняет version = "0.1.1" на 0.1.2.
   2. Заходит в OlliDesk/core/_version.py и меняет строку на __version__ = "0.1.2".
   3. Вызывает git-cliff. Тот сканирует историю коммитов (все ваши add, fix, upd), красиво группирует их и дописывает в CHANGELOG.md блок для версии v0.1.2.
   4. Делает git add для всех трех изменившихся файлов (pyproject.toml, _version.py, CHANGELOG.md).
   5. Делает автоматический релизный коммит.
   6. Ставит локальный Git-тег v0.1.2 на этот коммит.

## Шаг 3: Пуш в удаленный репозиторий

Вам остается только отправить результаты автоматизации на GitHub/GitLab:

git push origin main --tags

(Флаг --tags обязателен, чтобы на сервер ушел не только коммит релиза, но и сам созданный Git-тег)

По умолчанию git-cliff действительно работает как накопительный инструмент: он сканирует всю историю Git и генерирует ченджлог для всех найденных тегов (версий) сразу в один файл.

Если надо создать историю изменений только для одной конкретной версии (например, последней), есть два пути.

```sh
# Только для самого последнего тега:
git cliff --latest

# Для конкретного диапазона коммитов или тегов:
git cliff v1.0.0..v2.0.0

# Только для незакоммиченных изменений (unreleased):
git cliff --unreleased
```
