# Версии приложения

Фиксировать конкретную версию в обработчике ошибки — плохое решение. В активной разработке это значение быстро устареет, и вы снова запутаетесь.
Чтобы сразу видеть проблему в интерфейсе, лучше возвращать явную строку-индикатор ошибки, либо динамически считывать pyproject.toml «на лету», если пакет не установлен.
Вот три отличных варианта решения на выбор:

## Вариант 1. Информативная строка-индикатор (Самый простой)

Вместо фейковой версии возвращайте строку, которая сразу даст вам понять, что вы забыли сделать pip install -e . в текущем виртуальном окружении.

try:
    current_version = version("ollidesk")except PackageNotFoundError:
    # Сразу видно в окне "О программе", что окружение не настроено
    current_version = "unknown-dev (run 'pip install -e .')"

* Плюс: Идеально для дебага. Если в интерфейсе OlliDesk вы видите этот текст — вы сразу понимаете, что нужно запустить в терминале.

------------------------------

## Вариант 2. Динамическое чтение из pyproject.toml (Автоматический)

Если проект запущен из исходников, файл pyproject.toml всегда лежит в корне. Можно научить Python читать версию напрямую из него, если метаданные пакета недоступны.
Для этого в стандартной библиотеке Python (начиная с 3.11) есть модуль tomllib (в Python 3.8–3.10 используется библиотека toml).

import sysfrom pathlib import Pathfrom importlib.metadata import version, PackageNotFoundError
try:
    current_version = version("ollidesk")except PackageNotFoundError:
    # Пытаемся найти pyproject.toml относительно текущего файла app.py
    # OlliDesk/core/app.py -> поднимаемся на 2 уровня вверх до корня проекта
    pyproject_path = Path(__file__).resolve().parents[2] / "pyproject.toml"

    if pyproject_path.exists():
        # Для Python 3.11+ встроенный tomllib
        import tomllib
        with open(pyproject_path, "rb") as f:
            data = tomllib.load(f)
            current_version = data.get("project", {}).get("version", "0.0.0-path-err") + "-src"
    else:
        current_version = "0.0.0-unknown"

* Плюс: Полная автоматизация. Даже если вы создадите новое venv и запустите python main.py без установки, программа сама залезет в pyproject.toml и выведет, например, 0.1.1-src. Вы всегда будете видеть реальные цифры.

------------------------------

## Вариант 3. Использование модуля _version.py (Классический подход)

Многие крупные библиотеки (например, setuptools_scm) автоматически генерируют служебный файл_version.py во время сборки или бампа версии.
Если вы используете bump-my-version (из предыдущего шага), вы можете заставить его обновлять версию еще и в специальном маленьком файле OlliDesk/core/_version.py:

   1. Добавьте в pyproject.toml:

[[tool.bumpversion.files]]
filename = "OlliDesk/core/_version.py"
search = "__version__ = \"{current_version}\""
replace = "__version__ = \"{new_version}\""

   1. В самом файле OlliDesk/core/_version.py оставьте одну строку:

__version__ = "0.1.1"

   1. В app.py импортируйте её напрямую:

from OlliDesk.core._version import __version__# ...
self.app.setApplicationVersion(__version__)

* Плюс: Никаких try-except блоков, импорт работает всегда и везде, а bump-my-version гарантирует, что эта строка обновится одновременно с pyproject.toml и CHANGELOG.

Какой из вариантов (информативный unknown, авточтение pyproject.toml или автогенерация _version.py) кажется вам наиболее удобным для OlliDesk?

---

При Варианте 3 запуск через python main.py работает мгновенно и без задержек. Python просто импортирует обычную строковую переменную из файла _version.py, как любой другой ваш модуль. Никаких проверок окружения или чтения диска во время работы приложения не происходит.
Вот пошаговый алгоритм, как теперь будет выглядеть ваша работа и выпуск релизов.

## Подготовка (Делается один раз)

   1. Создайте файл OlliDesk/core/_version.py с текущей версией:

   __version__ = "0.1.1"

   2. В файле OlliDesk/core/app.py импортируйте её и передайте в Qt:

   from OlliDesk.core._version import __version__

# Внутри инициализации QApplication

   self.app.setApplicationVersion(__version__)

   3. Добавьте в pyproject.toml в секцию настроек bump-my-version отслеживание этого файла:

   [[tool.bumpversion.files]]
   filename = "OlliDesk/core/_version.py"
   search = "__version__ = \"{current_version}\""
   replace = "__version__ = \"{new_version}\""

------------------------------

## Ежедневная разработка (Как обычно)

Вы работаете в привычном режиме:

   1. Меняете код OlliDesk.
   2. Пишете коммиты с префиксами: fix: исправил баг в окне, add: добавил кнопку, upd: обновил логику.
   3. Делаете git add, git commit, git push.
   4. Запускаете python main.py для тестирования — приложение всегда видит текущую версию (например, 0.1.1).

------------------------------

## Шаг за шагом: Выпускаем обновление (Хотфикс или Минор)

Когда вы решили, что накопленные изменения пора зафиксировать в новый релиз, вместо привычной цепочки add -> commit -> push вы делаете следующее:

## Шаг 1: Запуск утилиты бампа

Прямо в терминале в корне проекта вы выполняете всего одну команду, в зависимости от типа изменений.

* Если это хотфикс / патч (версия 0.1.1 превратится в 0.1.2):

bump-my-version bump patch

* Если это минорное обновление (версия 0.1.1 превратится в 0.2.0):

bump-my-version bump minor

## Шаг 2: Что происходит автоматически за 1 секунду

Утилита bump-my-version сама делает всю рутину, которую раньше приходилось контролировать головой:

   1. Заходит в pyproject.toml и меняет version = "0.1.1" на 0.1.2.
   2. Заходит в OlliDesk/core/_version.py и меняет строку на __version__ = "0.1.2".
   3. Вызывает git-cliff. Тот сканирует историю коммитов (все ваши add, fix, upd), красиво группирует их и дописывает в CHANGELOG.md блок для версии v0.1.2.
   4. Делает git add для всех трех изменившихся файлов (pyproject.toml, _version.py, CHANGELOG.md).
   5. Делает автоматический релизный коммит.
   6. Ставит локальный Git-тег v0.1.2 на этот коммит.

## Шаг 3: Пуш в удаленный репозиторий

Вам остается только отправить результаты автоматизации на GitHub/GitLab:

git push origin main --tags

(Флаг --tags обязателен, чтобы на сервер ушел не только коммит релиза, но и сам созданный Git-тег)

------------------------------
